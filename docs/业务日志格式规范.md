# 业务日志格式规范

这份文档定义 Java 应用需要输出的最小结构化业务日志格式，用于当前 MVP 运维 Agent 识别工作流失败、卡住、Token 异常和工具调用失败。

## 目标

日志不需要一开始就非常完整，但至少要让运维 Agent 能回答下面几个问题：

- 哪个 workflow 失败了
- 卡在哪个节点
- 是 LLM 推理异常还是 Tool 调用异常
- 单次 workflow 是否出现异常高 Token 消耗
- 某类 Tool 是否在短时间内大量失败

## 输出要求

- 推荐输出到容器标准输出 `stdout`
- 每条业务事件一行 JSON
- 每条 JSON 都尽量包含 `event_type`
- 保持字段命名稳定，不要同义多套

## 最小字段

### 通用字段

- `event_type`: 事件类型
- `trace_id`: 链路追踪 ID
- `workflow_id`: 工作流 ID
- `node_id`: 节点 ID
- `status`: 当前状态
- `elapsed_ms`: 耗时，单位毫秒
- `error_code`: 错误码
- `error_message`: 错误描述
- `tool_name`: 工具名称
- `token_input`: 输入 Token 数
- `token_output`: 输出 Token 数
- `total_tokens`: 总 Token 数，可选但推荐
- `timestamp`: 业务事件时间戳

## 推荐事件类型

### workflow 状态事件

用于识别 workflow 是否卡住、失败率是否升高。

```json
{
  "event_type": "workflow_state",
  "timestamp": "2026-03-14T10:00:00Z",
  "trace_id": "trace-001",
  "workflow_id": "wf-1001",
  "node_id": "node-3",
  "status": "running",
  "elapsed_ms": 1200
}
```

### workflow 结果事件

用于识别 workflow 成功率与失败率。

```json
{
  "event_type": "workflow_result",
  "timestamp": "2026-03-14T10:00:05Z",
  "trace_id": "trace-001",
  "workflow_id": "wf-1001",
  "node_id": "node-3",
  "status": "failed",
  "elapsed_ms": 5300,
  "error_code": "LLM_TIMEOUT",
  "error_message": "upstream model timeout"
}
```

### workflow Token 使用事件

用于识别异常 Token 消耗。

```json
{
  "event_type": "workflow_usage",
  "timestamp": "2026-03-14T10:00:05Z",
  "trace_id": "trace-001",
  "workflow_id": "wf-1001",
  "status": "finished",
  "token_input": 1200,
  "token_output": 800,
  "total_tokens": 2000
}
```

### Tool 调用事件

用于识别外部 Tool 调用失败率。

```json
{
  "event_type": "tool_call",
  "timestamp": "2026-03-14T10:00:03Z",
  "trace_id": "trace-001",
  "workflow_id": "wf-1001",
  "node_id": "node-3",
  "tool_name": "search_api",
  "status": "failed",
  "elapsed_ms": 2300,
  "error_code": "AUTH_EXPIRED",
  "error_message": "token expired"
}
```

## 状态值建议

为了让探测逻辑保持简单，建议状态值尽量收敛为以下集合：

- `running`
- `success`
- `failed`
- `error`
- `cancelled`

## 当前 MVP 已识别规则

当前代码会直接利用这些字段做以下判断：

- `workflow_state + running + elapsed_ms` 识别 workflow 卡住
- `workflow_result` 识别失败率升高
- `workflow_usage` 识别 Token 异常
- `tool_call` 识别 Tool 失败率升高

## Java 落地建议

- 如果你用的是 Spring Boot，建议在工作流主链路中统一封装一层结构化日志方法
- 不要把 JSON 日志拆成多行
- 不要在一条 JSON 前面再拼过长文本前缀
- 如果必须保留文本前缀，也要保证后面仍然是完整 JSON 对象

## 一句话建议

先保证事件字段稳定，再追求字段完整。对当前 MVP 来说，稳定比全面更重要。
