三层具体监控什么，SRE Agent 怎么处置
第一层：基础设施（2核2G 是瓶颈，重点盯）
2核2G 资源很紧张，这层最容易出问题：

CPU 突刺：Agent 推理调用 LLM 时 CPU 会飙，需要区分是正常推理负载还是死循环
内存泄漏：Java 应用长时间运行堆内存增长，触发 Full GC，最终 OOM
磁盘打满：工作流执行日志、用户数据持续写入，2G 磁盘很容易满

SRE Agent 处置：发现磁盘 > 85% 自动清理 30 天前的日志，发现 OOM 自动重启容器并保存 heapdump。

第二层：Java 应用（最有技术含量）
这是你最熟悉也最能讲故事的一层：

接口响应时间（RT）：正常 < 200ms，如果工作流编排接口 RT 突然涨到 2s，说明有问题
GC 频率异常：Minor GC 每秒超过 5 次，或出现 Full GC，说明堆内存压力大
ERROR 日志突增：用正则扫描 log 文件，统计每分钟 ERROR 数量，突增即告警
线程池打满：工作流并发高时线程池队列积压，导致请求超时

SRE Agent 处置：拉 jstack 分析线程状态，定位是哪个工作流节点阻塞，输出具体代码位置。

第三层：业务逻辑（你的平台独有，这是差异化）
这一层是你相比 fuzzylabs 原版最大的创新点，因为原版完全没有业务层监控：

工作流执行成功率：用户创建的 workflow 跑失败了，是哪个节点失败的，是 LLM 超时还是工具调用报错
Agent 推理 Token 消耗异常：某个用户的 workflow 单次消耗 50k token，可能是 prompt 设计有问题或进入了推理死循环
MCP Tool 调用失败：用户配置了外部工具（比如调用第三方 API），失败率突增，自动诊断是网络问题还是鉴权过期
工作流卡住不动：某个 workflow 状态一直是 running，超过 5 分钟没有状态变更，自动判定为死锁并终止
